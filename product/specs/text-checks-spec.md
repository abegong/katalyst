# Text checks: regex and string policy over body content

> **Status: planning.** Adds text rules that run regex/string-policy checks
> directly against an item's body (and against plain-text files with no
> frontmatter), with line-level diagnostics and optional auto-fixes. Resolves
> issue #49. **Open structural question:** whether these ship as a new fourth
> `text` family or are arranged some other way — parked for discussion (Open
> Question 1); the rest of the design is independent of it.

## Overview

Katalyst checks today reach frontmatter (the `object` family), markdown
structure (the `markdown` family), and filename/path conventions (the
`filesystem` family). Nothing lints the **body prose** as raw text: "no trailing
period on the title line," "must contain the words *Generated by*," "never write
*TODO* in a published note." These are presence/absence rules over text, and the
only check that touches the body — the markdown family — reasons about headings
and fences, not arbitrary line shapes. This spec adds three rules —
`text_requires` and `text_forbids` (regex), and a `text_denylist` of literal
substrings — that target the body directly through a span selector (`body`,
lines, lines matching a regex), with line-level diagnostics and an optional
declared **fix**.

## Value

- **Prose constraints become first-class.** Style rules ("no trailing
  whitespace," "title line has no period," "forbidden phrasing") are the rules
  authors hit first in a writing vault, and today they need external linters or
  brittle wrappers. One documented surface (`pattern`, `target`) covers them.
- **Plain-text files are in scope, not just frontmatter-bearing markdown.** The
  parser already returns the entire file as `Body` when there is no frontmatter
  (`frontmatter.go:138`), so a collection of `.txt` or frontmatter-less `.md`
  files can be linted the same way — the rule just needs to read `Body`.
- **Fixable where it's safe.** Many prose violations (a stray trailing period, a
  forbidden synonym) are a mechanical search-and-replace. Letting a rule declare
  its own fix makes `katalyst fix` repair them, extending `fix` from a
  frontmatter formatter to a body repairer **only where the author opts in**.

## Current State

- **Three families, none reads raw body text.** `internal/checks/registry.go`
  defines `objects`, `markdown`, and `filesystem` families (`Families()`).
  `object`/`filesystem` never look at the body; the `markdown` family does, but
  only through structure-aware helpers (`firstH1`, `heading`, fence scanning in
  `internal/checks/markdown.go`).
- **The body is already available, line-numbered.** Every check receives
  `Context{Doc *frontmatter.Document, …}` (`internal/checks/checks.go:6`).
  `Doc.Body []byte` holds the body and `Doc.BodyLine` is the 1-based line where
  the body starts (`frontmatter.go:100`). The markdown family already splits the
  body into `(line, text)` pairs via `markdownLines(body, bodyLine)`
  (`markdown.go:171`); text rules reuse that exact helper for per-line
  diagnostics. For a file with no frontmatter, `Body` is the whole file and
  `BodyLine` is 1 (`frontmatter.go:138`), so line numbers stay correct.
- **A regex check type already exists, but anchored and over the filename.**
  `filesystem_name_regex` compiles `^(?:pattern)$` and tests the basename
  (`config.go:792`, `engine.go:156`). Its anchoring is wrong for prose: text
  rules are about a pattern appearing (or not) *somewhere* in a span, not the
  span matching exactly. Text rules reuse the `pattern` key and the
  `AnchoredPattern` precedent but compile **unanchored**.
- **`fix` is a frontmatter formatter that refuses to invent.** `cmd/fix.go`
  canonicalizes frontmatter (sorted keys, block style, one trailing newline) and
  **preserves the body verbatim**; it "never invents semantic values." A
  rule-declared body fix is a deliberate, scoped extension of this contract — see
  Design.
- **Four-touchpoint wiring, parity-enforced.** A new check type is a constant +
  `normalizeCheck` case in `internal/config/config.go`, a `Run` in
  `internal/checks/`, a dispatch case in `cmd/engine.go` (`checksFor`), and a
  `Descriptor` in `registry.go`; `registry_test.go` fails the build if a
  dispatched type has no Descriptor, and `cmd/gendocs` renders
  `docs/content/reference/check-types/` from the Descriptors.
- **Checks are per-item.** Text rules need no sibling context, so they implement
  the existing per-item `Check.Run(ctx) []Violation` contract.

## Design

### Family placement (parked)

Where these rules live in the docs taxonomy — a new fourth `text` family, or
some other arrangement — is **Open Question 1** and is being discussed
separately. The rest of this Design (span selector, the three rules, fix) holds
regardless of that answer; the family decision changes only the `Family` value in
the registry and the reference subdirectory the pages render into. The spec
provisionally writes them as a `text` family so the examples read concretely.

### The span selector: `target`

Every text rule is evaluated against a set of **spans** — slices of body text —
chosen by `target` (default `body`). A rule with one span (`body`, `first-line`)
is a single assertion; a rule whose target yields many spans (`line`,
`matched-lines`) is evaluated per span.

| `target` | Spans produced |
|---|---|
| `body` | one span: the entire body as a multiline string |
| `line` | one span per body line |
| `first-line` | one span: the first non-empty body line |
| `matched-lines` (+ `select: <regex>`) | one span per body line matching `select` |

`first-line` skips leading blank lines so an "opening line" rule is not defeated
by a blank line after the frontmatter fence. `select` is valid only with
`matched-lines`. This is the **lean v1 selector**; richer targets — `last-line`,
`line-number` (nth line), `paragraph`, and an `offset` modifier for the nth line
before/after a matched line — are deferred to **issue #51**. They are purely
additive (they extend the `target` enum and share the same `spans()` helper) and
change nothing in the rule model.

### The three rules

Each rule matches against every span the `target` selects. Two are regex-based;
one is a literal denylist.

- **`text_requires`** — a `pattern` (Go regex, compiled **unanchored** — the
  deliberate divergence from `filesystem_name_regex`, whose `^…$` anchoring is
  wrong for prose) must appear, with a `match` knob (default `any`) that only
  does work for multi-span targets:

  | `match` | Passes when |
  |---|---|
  | `any` (default) | at least one selected span matches — "the body mentions *Sources*" |
  | `all` | every selected span matches — "every heading line is Title Case" |

  For single-span targets (`body`, `first-line`) `any` and `all` coincide.
  `match: any | all` is spelled as two self-describing values rather than a
  boolean so neither the field nor a default needs decoding.

- **`text_forbids`** — a `pattern` (same unanchored regex) must appear in **no**
  span. "No line contains TODO" is `kind: text_forbids, target: line, pattern:
  '\bTODO\b'`. It needs no `match` knob: "forbid X" only ever means "X is in no
  span."

- **`text_denylist`** — forbids a **list of literal substrings** (`values:`),
  each matched with metacharacters inert (via `regexp.QuoteMeta`). `values:
  [TODO, FIXME, XXX]` bans all three in one check — the ergonomic form of a
  word-ban that a regex alternation (`TODO|FIXME|XXX`) expresses awkwardly and
  with escaping hazards. It honors `target` like the regex rules and is
  report-only (no `fix`; reach for `text_forbids` with a `fix` when a literal
  needs rewriting). This is the dedicated list kind chosen in Open Question 3;
  its **name is Open Question 2**.

Diagnostics name the failing span(s): `text_forbids`/`text_denylist` report each
matching span (denylist names the offending value); `text_requires match: all`
reports each non-matching span; `text_requires match: any` reports one body-level
violation when nothing matched.

### Optional fix

Only `text_forbids` may declare a `fix` — a replacement template applied to the
matched (forbidden) text with `regexp` capture-group syntax (`$1`, `${name}`):

```yaml
- kind: text_forbids
  target: first-line
  pattern: '\.(\s*)$'   # a period (+ trailing space) ending the first line
  fix: '$1'             # drop the period, keep any trailing whitespace
```

The asymmetry is principled. `text_forbids` always **locates** the offending
text — that is exactly what its pattern matched — so a fix has a concrete span to
rewrite. `text_requires` has nothing to locate: a missing required phrase has no
position, and inventing one is the value-invention `fix` already refuses
(`cmd/fix.go`); so a `fix` on `text_requires` is a config error.
`text_denylist` is report-only by design (above). Further:

- **A fix replaces only the matched substring** within the span, never the whole
  span or file, so unrelated text is untouched. The replacement is purely
  mechanical (regex substitution), like the rest of `fix`.
- **`fix` is opt-in.** `text_forbids` rules without `fix` stay report-only,
  exactly as today; the `fix` command's body-preservation default is unchanged
  for them.
- **The fix re-checks its own work.** After applying the replacement, the engine
  re-runs the rule against the rewritten body; if the span still violates (a bad
  template — e.g. the replacement re-introduces the forbidden pattern), `fix`
  fails with "fix did not resolve the violation" rather than silently writing a
  still-broken file. The item is already in memory, so the extra pass is cheap.

The engine applies the replacement generically from the config (pattern +
replacement + span), so no per-check-type fix code is needed.

### Diagnostics

Every violation carries the file and a 1-based line number.

- Line spans report their own line (`BodyLine + index`, as `markdownLines`
  already computes).
- For `target: body`, compute the line by counting newlines in `Body` up to the
  match's start offset and adding `BodyLine`. The `Path` field stays `/`
  (body-scoped), matching how the markdown family reports body violations
  (`markdown.go:36`).

Messages follow the `path:line: message` grammar: `forbidden text /\bTODO\b/
found`, `required text /Sources/ not found`, `forbidden term "FIXME" found`,
quoting the pattern with regex delimiters (and a denylist value with quotes).

### Config surface

`internal/config/config.go`:

- Add `CheckTextRequires`, `CheckTextForbids`, and `CheckTextDenylist` constants.
- Extend `rawCheck`/`CheckInstance` with `Match string` (requires only),
  `Select string`, and `Fix string` (forbids only). Reuse `Pattern` and `Target`
  (regex rules) and the existing `Values []string` (denylist).
- `normalizeCheck`:
  - `text_requires`/`text_forbids`: require `pattern`; compile unanchored to
    reject bad regexes at load time (as `filesystem_name_regex` does eagerly,
    `config.go:796`).
  - `text_denylist`: require a non-empty `values`.
  - validate `target`; require `select` iff `matched-lines` and reject it
    otherwise; validate `match ∈ {any, all}` and reject it on
    `text_forbids`/`text_denylist`; reject `fix` on
    `text_requires`/`text_denylist`.

`internal/checks/text.go` (new): `TextRequires`/`TextForbids`/`TextDenylist`
structs sharing a `spans(ctx) []span` helper (each span carries text + start
line) and compiled regexes. `cmd/engine.go` dispatches them; `cmd/fix.go` gains a
body pass that applies declared replacements and re-checks. `registry.go` gets
the Descriptors and, per Open Question 1, possibly a fourth `Family`.

## Open Questions

### 1. Family placement (structural; parked for discussion)

**Context.** Check types are grouped into *families* — `objects`, `markdown`,
`filesystem` — defined by `Families()` in `internal/checks/registry.go`. A
family fixes the docs subdirectory (`reference/check-types/<family>/`), the intro
copy, and how `katalyst check-types list` groups output. Text rules need a home.
The `markdown` family already reads the body, but only through markdown-*aware*
helpers (headings, fences); text rules are structure-agnostic and also apply to
non-markdown `.txt` files.

**Choices & tradeoffs.**

| Option | Buys | Costs |
|---|---|---|
| New `text` family | A truthful intro ("raw body content, independent of markdown structure"); honest that rules apply to `.txt` | A fourth family is more surface for users to learn and docs to maintain |
| Fold into `markdown` | No new family; all body checks in one place | Forces the markdown intro to also mean "arbitrary regex over raw text," blurring markdown-aware vs structure-agnostic, and misreads for `.txt` |
| Rename to a broader family (`content`) spanning markdown + text | One body-content family | Renames existing markdown pages/kinds — real churn for a taxonomy bet |

**Recommendation.** New `text` family: the structure-agnostic vs markdown-aware
distinction is real and keeps each family intro true. Your call.

### 2. Name for the literal-denylist kind

**Context.** Open Question 3 settled on a dedicated kind that forbids a list of
literal substrings (regex metacharacters inert). The spec provisionally calls it
`text_denylist`; the earlier draft's `text_forbids_substrings` was rejected as
clunky. The name should read well next to `text_requires`/`text_forbids` and the
existing `*_in` / `*_enum` membership kinds, and signal "a set of literal strings
that must not appear."

**Choices & tradeoffs.**

| Candidate | Reads as | Concern |
|---|---|---|
| `text_denylist` (provisional) | "a denylist of strings" | "denylist" implies an "allowlist" counterpart that won't exist |
| `text_forbidden_terms` | "these terms are forbidden" | "terms" hints at words, not arbitrary substrings |
| `text_banned_phrases` | "these phrases are banned" | "phrases" likewise leans word/phrase, not substring; new verb ("banned") |
| `text_forbids` with a `values:` list (no new kind) | reuse one kind for regex *and* literal sets | overloads one kind with two matchers (`pattern` xor `values`); muddier validation and docs |

**Recommendation.** `text_denylist` — shortest and most precise about "a set
that must not appear," and the missing allowlist counterpart hasn't caused
confusion for similar names elsewhere. Easy to rename before code lands. Your
call.

- **Generated reference** — add the `Descriptor`s (and, per OQ1, a `text`
  `Family`) in `internal/checks/registry.go`, then `make docs-gen` to render the
  reference page(s) (do not hand-edit). New kinds then appear in `katalyst
  check-types list`; the `registry_test.go` parity test enforces they ship
  documented.
- **User docs (Hugo)** — `docs/content/how-to/configure-rules.md` gains text-rule
  examples (forbid `TODO` via `text_denylist`; ban a trailing period on the first
  line with a `fix`). `docs/content/reference/configuration.md` documents the
  span selector, the `match` knob, the unanchored-regex limitation, the opt-in
  `fix` and its re-check, and that text rules lint frontmatter-less files when the
  collection `pattern` matches them.
- **Glossary** — `docs/content/reference/glossary.md` gains "text rule" and
  "span" entries; the existing **Body** entry supplies the target term.
- **Developer docs** — root `AGENTS.md` records the span-selector model, the
  unanchored-regex divergence from `filesystem_name_regex`, and the opt-in
  body-fix extension to the `fix` contract (which `cmd/fix.go`'s "never invents"
  note must now cross-reference). `internal/checks` package docs cover the
  `spans` helper.

## Deferred (tracked separately)

- **Richer span targets** (`last-line`, `line-number`/nth line, `paragraph`,
  `offset` relative to a matched line) — **issue #51**. Lean v1 ships
  `body`/`line`/`first-line`/`matched-lines`.
- **Inline examples on check types** (pass/fail samples that document and
  self-test a rule, including its fix) — **issue #52**. A cross-cutting mechanism
  beyond the text family; out of scope here.

## Test checklist (what the pending tests assert)

Span selector:
- [ ] `body` matches across newlines (multiline pattern)
- [ ] `line` evaluates each line; `first-line` skips leading blanks
- [ ] `matched-lines` + `select` evaluates only matching lines
- [ ] `select` is rejected unless `target: matched-lines`; `matched-lines`
      requires `select`

Rules:
- [ ] `text_forbids` fails when the pattern is in any span; passes when absent
- [ ] `text_requires match: any` passes on one match; `match: all` requires every
      span; `any`==`all` for `body`/`first-line`
- [ ] `match` is rejected on `text_forbids`/`text_denylist`
- [ ] `text_denylist` flags any listed literal; metacharacters are inert;
      requires a non-empty `values`
- [ ] patterns are unanchored; an invalid `pattern` is a load-time error
- [ ] `body` violations report the match's line; line spans report their own
      line; a frontmatter-less file lints with lines from 1

Fix:
- [ ] a declared `fix` rewrites only the matched substring
- [ ] after a fix, the rule is re-checked; a template that leaves the span
      violating fails with "fix did not resolve the violation"
- [ ] `fix` on `text_requires`/`text_denylist` is a load error
- [ ] `text_forbids` without `fix` stays report-only; `fix` still preserves body

Registry / docs:
- [ ] every new kind has a Descriptor (parity stays green)
- [ ] `make docs-gen` renders the reference cleanly
