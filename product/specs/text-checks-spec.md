# Text checks: regex and string policy over body content

> **Status: planning.** Adds text rules that run regex/string-policy checks
> directly against an item's body (and against plain-text files with no
> frontmatter), with line-level diagnostics, optional auto-fixes, and inline
> examples. Resolves issue #49. **Open structural question:** whether these ship
> as a new fourth `text` family or are arranged some other way â€” parked for
> discussion (Open Question 1); the rest of the design is independent of it.

## Overview

Katalyst checks today reach frontmatter (the `object` family), markdown
structure (the `markdown` family), and filename/path conventions (the
`filesystem` family). Nothing lints the **body prose** as raw text: "no trailing
period on the title line," "must contain the words *Generated by*," "never write
*TODO* in a published note." These are presence/absence rules over text, and the
only check that touches the body â€” the markdown family â€” reasons about headings
and fences, not arbitrary line shapes. This spec adds text rules that target the
body directly: a span selector (`body`, lines, paragraphs, lines relative to a
match), a regex/substring matcher with an `all`/`any`/`none` quantifier, an
optional declared **fix**, and inline **examples** that double as documentation
and self-tests.

## Value

- **Prose constraints become first-class.** Style rules ("no trailing
  whitespace," "title line has no period," "forbidden phrasing") are the rules
  authors hit first in a writing vault, and today they need external linters or
  brittle wrappers. One documented surface (`pattern`, `target`, `quantifier`)
  covers them.
- **Plain-text files are in scope, not just frontmatter-bearing markdown.** The
  parser already returns the entire file as `Body` when there is no frontmatter
  (`frontmatter.go:138`), so a collection of `.txt` or frontmatter-less `.md`
  files can be linted the same way â€” the rule just needs to read `Body`.
- **Fixable where it's safe.** Many prose violations (a stray trailing period, a
  forbidden synonym) are a mechanical search-and-replace. Letting a rule declare
  its own fix makes `katalyst fix` repair them, extending `fix` from a
  frontmatter formatter to a body repairer **only where the author opts in**.
- **Examples make rules legible and self-testing.** A rule that ships positive
  and negative examples documents itself and can be verified â€” including that its
  fix turns each negative example into the stated result.

## Current State

- **Three families, none reads raw body text.** `internal/checks/registry.go`
  defines `objects`, `markdown`, and `filesystem` families (`Families()`).
  `object`/`filesystem` never look at the body; the `markdown` family does, but
  only through structure-aware helpers (`firstH1`, `heading`, fence scanning in
  `internal/checks/markdown.go`).
- **The body is already available, line-numbered.** Every check receives
  `Context{Doc *frontmatter.Document, â€¦}` (`internal/checks/checks.go:6`).
  `Doc.Body []byte` holds the body and `Doc.BodyLine` is the 1-based line where
  the body starts (`frontmatter.go:100`). The markdown family already splits the
  body into `(line, text)` pairs via `markdownLines(body, bodyLine)`
  (`markdown.go:171`); text rules reuse that exact helper for per-line
  diagnostics. For a file with no frontmatter, `Body` is the whole file and
  `BodyLine` is 1 (`frontmatter.go:138`), so line numbers stay correct.
- **A regex check type already exists, but anchored and over the filename.**
  `filesystem_name_regex` compiles `^(?:pattern)$` and tests the basename
  (`config.go:792`, `engine.go:156`). Its anchoring is wrong for prose: text
  rules are about a pattern appearing (or not) *somewhere* in a span, not the
  span matching exactly. Text rules reuse the `pattern` key and the
  `AnchoredPattern` precedent but compile **unanchored**.
- **`fix` is a frontmatter formatter that refuses to invent.** `cmd/fix.go`
  canonicalizes frontmatter (sorted keys, block style, one trailing newline) and
  **preserves the body verbatim**; it "never invents semantic values." A
  rule-declared body fix is a deliberate, scoped extension of this contract â€” see
  Design.
- **Four-touchpoint wiring, parity-enforced.** A new check type is a constant +
  `normalizeCheck` case in `internal/config/config.go`, a `Run` in
  `internal/checks/`, a dispatch case in `cmd/engine.go` (`checksFor`), and a
  `Descriptor` in `registry.go`; `registry_test.go` fails the build if a
  dispatched type has no Descriptor, and `cmd/gendocs` renders
  `docs/content/reference/check-types/` from the Descriptors.
- **Checks are per-item.** Text rules need no sibling context, so they implement
  the existing per-item `Check.Run(ctx) []Violation` contract.

## Design

### Family placement (parked)

Where these rules live in the docs taxonomy â€” a new fourth `text` family, or
some other arrangement â€” is **Open Question 1** and is being discussed
separately. The rest of this Design (span selector, quantifier, fix, examples)
holds regardless of that answer; it describes one or two new check *types*, and
the only thing the family decision changes is their `Family` value in the
registry and the reference subdirectory they render into. The spec provisionally
writes them as a `text` family so the examples read concretely.

### The span selector: `target`

Every text rule is evaluated against a set of **spans** â€” slices of body text â€”
chosen by `target` (default `body`). A rule with one span (`body`, `first-line`)
is a single assertion; a rule whose target yields many spans (`line`,
`paragraph`, `matched-lines`) is evaluated per span, and the quantifier (below)
says how many must hold.

| `target` | Spans produced |
|---|---|
| `body` | one span: the entire body as a multiline string |
| `line` | one span per body line |
| `first-line` / `last-line` | one span: the first/last non-empty body line |
| `line-number` (+ `index: N`) | one span: the Nth body line (1-based; negative counts from the end) |
| `paragraph` | one span per blank-line-delimited block (multiline) |
| `matched-lines` (+ `select: <regex>`) | one span per body line matching `select` |

Two modifiers compose with the line targets:

- **`index: N`** picks the Nth span for the multi-span targets â€” e.g.
  `target: paragraph, index: 1` is the first paragraph; `target: line-number,
  index: -1` is the last line.
- **`offset: N`** combines with `matched-lines` to select the line *N* lines
  after (negative = before) each matched line instead of the matched line
  itself. "The line after every heading" is `target: matched-lines, select: '^#
  ', offset: 1`. This is the "nth line before/after a matched line" capability:
  it lets a rule about a *related* line (the line under a heading, the line
  before a fence) be written without the family learning markdown structure.

`first-line` skips leading blank lines so an "opening line" rule is not defeated
by a blank line after the frontmatter fence. `select` and `offset` are valid only
with `matched-lines`; `index` only with `line-number`/`paragraph`/etc.

### The matcher: `pattern`, `literal`, `quantifier`

A rule matches `pattern` (a Go regex, compiled **unanchored** â€” the deliberate
divergence from `filesystem_name_regex`) against each span. `literal: true`
wraps `pattern` in `regexp.QuoteMeta` so it matches as a plain substring,
covering the issue's "string-policy" half.

`quantifier` (default `all`) says how many spans must contain a match:

| `quantifier` | Passes when |
|---|---|
| `all` | every selected span matches (e.g. every heading line is Title Case) |
| `any` | at least one selected span matches (e.g. the body mentions *Sources*) |
| `none` | no selected span matches (e.g. no line contains *TODO*) |

For single-span targets (`body`, `first-line`) `all` and `any` coincide. This
collapses what were two kinds (`text_requires` / `text_forbids`) into one
parameterized **`text_match`** kind: `requires` is `any`, `forbids` is `none`,
and `all` (every line/paragraph must conform) is newly expressible. (Resolving
the old "any vs all" question â€” see Open Question 2.) Violation diagnostics name
the failing span(s): `none` reports each matching span; `all` reports each
non-matching span; `any` reports one document-level violation when nothing
matched.

### Optional fix

A rule may declare a `fix` â€” a replacement template applied to the matched text
with `regexp` capture-group syntax (`$1`, `${name}`). When `katalyst fix` runs
and a span violates the rule, katalyst rewrites the offending match in the body
using `fix`:

```yaml
- kind: text_match
  target: first-line
  pattern: '\.(\s*)$'     # a period (+ trailing space) at end of the first line
  quantifier: none
  fix: '$1'               # drop the period, keep any trailing whitespace
```

Constraints, so this stays inside the spirit of "never invent":

- **Fixes attach to `none`/`all` rules whose `pattern` locates the offending
  text.** An `any` "must mention X" rule has nothing to locate and no safe
  insertion point, so a fix on it is a config error â€” auto-inserting required
  prose is exactly the value-invention `fix` refuses (`cmd/fix.go`).
- **A fix replaces only the matched substring** within the span, never the whole
  span or file, so unrelated text is untouched. The replacement is purely
  mechanical (regex substitution), like the rest of `fix`.
- **`fix` is opt-in per rule.** Rules without `fix` stay report-only, exactly as
  today; the `fix` command's body-preservation default is unchanged for them.

The engine applies the replacement generically from the config (pattern +
replacement + span), so no per-check-type fix code is needed. Whether a declared
fix can leave the document still-violating (a bad template) is validated by the
rule's own examples (next) and re-checked after applying â€” **Open Question 4**.

### Examples

A rule may carry `examples` â€” positive and negative samples that document it and
are verified at test/CI time (and optionally at load time):

```yaml
- kind: text_match
  target: first-line
  pattern: '\.(\s*)$'
  quantifier: none
  fix: '$1'
  examples:
    pass:
      - "My Note Title"
    fail:
      - text: "My Note Title."
        fixed: "My Note Title"      # required iff the rule declares a fix
```

Verification asserts: every `pass` example satisfies the rule; every `fail`
example violates it; and when the rule declares a `fix`, applying it to each
`fail.text` yields `fail.fixed` (and the result then passes). This makes examples
executable documentation â€” the same samples render into the generated reference
page and back a `katalyst check-types verify` pass (name TBD). The mechanism is
generic over check types, not text-specific, so it can later cover the other
families; this spec only commits to wiring it for text rules. How far to take it
in v1 â€” load-time enforcement vs a separate verify command, and whether examples
are required â€” is **Open Question 5**.

### Diagnostics

Every violation carries the file and a 1-based line number.

- Line/paragraph spans report their own start line (`BodyLine + index`, as
  `markdownLines` already computes).
- For `target: body`, compute the line by counting newlines in `Body` up to the
  match's start offset and adding `BodyLine`. The `Path` field stays `/`
  (body-scoped), matching how the markdown family reports body violations
  (`markdown.go:36`).

Messages follow the `path:line: message` grammar: `forbidden text /\bTODO\b/
found`, `required text /Sources/ not found`, quoting the pattern with regex
delimiters.

### Config surface

`internal/config/config.go`:

- Add `CheckTextMatch` (and a second constant only if Open Question 2 keeps two
  kinds).
- Extend `rawCheck`/`CheckInstance` with `Quantifier string`, `Select string`,
  `Index *int`, `Offset *int`, `Literal bool`, `Fix string`, and an `Examples`
  block (`pass []string`, `fail []{text, fixed}`). `Pattern` and `Target` are
  reused.
- `normalizeCheck`: require `pattern`; compile unanchored to reject bad regexes
  at load time (as `filesystem_name_regex` does eagerly, `config.go:796`);
  validate `target` and `quantifier` enums; require `select` iff `matched-lines`
  and reject `offset`/`select`/`index` outside their targets; reject `fix` on an
  `any` rule; require `fail.fixed` iff `fix` is set.

`internal/checks/text.go` (new): a `TextMatch` struct with a `spans(ctx) []span`
helper (each span carries text + start line) and a compiled `*regexp.Regexp`;
`Run` applies the quantifier. `cmd/engine.go` dispatches it; `cmd/fix.go` gains a
body pass that applies declared replacements. `registry.go` gets the
Descriptor(s) and, per Open Question 1, possibly a fourth `Family`.

## Open Questions

1. **Family placement (structural; parked for discussion).** Do text rules form
   a new fourth `text` family, fold into `markdown`, or get arranged some other
   way? The markdown family is markdown-*aware* (headings, fences); text rules
   are structure-agnostic and apply to `.txt`, which argues for a separate
   family â€” but a fourth family is also more surface to learn. This changes only
   the registry `Family` value and the docs subdirectory, not the rule design.
   *To resolve with the user.*

2. **One kind or two.** The Design collapses `text_requires`/`text_forbids` into
   one `text_match` kind with `quantifier: all|any|none`. The only downside of
   the single-kind form: `quantifier` does real work only for the multi-span
   targets (`line`, `paragraph`, `matched-lines`); for `body`/`first-line`,
   `all` and `any` are identical, so the knob is occasionally a no-op the user
   must still understand. Two kinds (`requires` = any, `forbids` = none) read
   more like English at the call site but can't express `all` (every line
   conforms) without a third kind. Recommendation: **one kind**, since it is
   strictly more expressive and `all` is a real ask ("every heading is Title
   Case"). *Leaning resolved; confirm.*

3. **`literal` flag vs a list-valued substring kind.** Plain-string matching can
   be a `literal: true` boolean on `text_match` (current Design), where banning a
   *set* of words is written as a regex alternation (`TODO|FIXME|XXX`); or a
   dedicated kind taking a literal **list** (`substrings: [TODO, FIXME, XXX]`),
   which reads better for word-bans but duplicates the rule axis and needs its
   own fix/examples story. Recommendation: ship the `literal` flag and lean on
   regex alternation for sets; add a list form only if authors ask. *Confirm.*

4. **Fix safety re-check.** After applying a declared `fix`, should the engine
   re-run the rule and error if the document still violates (catching a bad
   replacement template), or trust the rule's `fail.fixed` examples to have
   caught it offline? Recommendation: re-check after fixing and surface a
   "fix did not resolve the violation" error, since a silent no-op fix is worse
   than a frontmatter-only `fix`. *Open.*

5. **Examples scope in v1.** Are `examples` required or optional; verified only by
   a `check-types verify` command / CI, or also enforced at config load time
   (every example must hold before the rule will run); and text-only or wired
   generically for all families now? Recommendation: optional, verified by an
   explicit command + CI (not load time, to keep `check` fast), and built generic
   but only documented for text rules in v1. *Open.*

## Test checklist (what the pending tests assert)

Span selector:
- [ ] `body` matches across newlines (multiline pattern)
- [ ] `line` evaluates each line; `first-line`/`last-line` skip blanks
- [ ] `line-number` with `index: N` (incl. negative) selects the right line
- [ ] `paragraph` splits on blank lines; `index` picks the Nth
- [ ] `matched-lines` + `select` selects matching lines; `offset` shifts to the
      line N before/after each match
- [ ] modifier misuse is a load error (`select`/`offset` off `matched-lines`,
      `index` off an indexed target)

Matcher:
- [ ] `quantifier: all|any|none` pass/fail per the table; `all`==`any` for `body`
- [ ] `literal: true` matches metachars inertly; patterns are unanchored
- [ ] an invalid `pattern` is a load-time error
- [ ] `body` violations report the match's line; line/paragraph spans report
      their own line; a frontmatter-less file lints with lines from 1

Fix:
- [ ] a declared `fix` rewrites only the matched substring; document re-checks
      clean (or errors per OQ4)
- [ ] `fix` on an `any` rule is a load error
- [ ] rules without `fix` stay report-only; `fix` still preserves their body

Examples:
- [ ] each `pass` example satisfies the rule; each `fail` violates it
- [ ] when `fix` is set, applying it to `fail.text` yields `fail.fixed`; `fixed`
      is required iff `fix` is set

Registry / docs:
- [ ] every new kind has a Descriptor (parity stays green)
- [ ] `make docs-gen` renders the reference cleanly
