# Text checks: regex and string policy over body content

> **Status: planning.** Adds a fourth check family, `text`, that runs regex and
> string-policy rules directly against an item's body (and against plain-text
> files with no frontmatter), with line-level diagnostics. Resolves issue #49.
> The markdown family stays markdown-*aware* (headings, fences, H1); the text
> family is structure-agnostic — it sees raw lines and the whole body string.

## Overview

Katalyst checks today reach frontmatter (the `object` family), markdown
structure (the `markdown` family), and filename/path conventions (the
`filesystem` family). Nothing lints the **body prose** as raw text: "no trailing
period on the title line," "must contain the words *Generated by*," "never write
*TODO* in a published note." These are presence/absence rules over text, and the
only check that touches the body — the markdown family — reasons about headings
and fences, not arbitrary line shapes. This spec adds a `text` family of
regex/substring rules that target the body directly, report the offending line,
and run through the existing `katalyst check` flow.

## Value

- **Prose constraints become first-class.** Style rules ("no trailing
  whitespace," "title line has no period," "forbidden phrasing") are the rules
  authors hit first in a writing vault, and today they need external linters or
  brittle wrappers. One documented surface (`pattern`, `target`) covers them.
- **Plain-text files are in scope, not just frontmatter-bearing markdown.** The
  parser already returns the entire file as `Body` when there is no frontmatter
  (`frontmatter.go:138`), so a collection of `.txt` or frontmatter-less `.md`
  files can be linted the same way — the family just needs to read `Body`.
- **Completes the body-content story the filesystem spec deferred.** That spec
  noted "the markdown layer does not parse links" and split body concerns out as
  future work (`filesystem-checks-spec.md`); text rules are the structure-
  agnostic half of that story.

## Current State

- **Three families, none reads raw body text.** `internal/checks/registry.go`
  defines `objects`, `markdown`, and `filesystem` families (`Families()`).
  `object`/`filesystem` never look at the body; the `markdown` family does, but
  only through structure-aware helpers (`firstH1`, `heading`, fence scanning in
  `internal/checks/markdown.go`).
- **The body is already available, line-numbered.** Every check receives
  `Context{Doc *frontmatter.Document, …}` (`internal/checks/checks.go:6`).
  `Doc.Body []byte` holds the body and `Doc.BodyLine` is the 1-based line where
  the body starts (`frontmatter.go:100`). The markdown family already splits the
  body into `(line, text)` pairs via `markdownLines(body, bodyLine)`
  (`markdown.go:171`); a text family can reuse that exact helper for per-line
  diagnostics. For a file with no frontmatter, `Body` is the whole file and
  `BodyLine` is 1 (`frontmatter.go:138`), so line numbers stay correct.
- **A regex check type already exists, but anchored and over the filename.**
  `filesystem_name_regex` compiles `^(?:pattern)$` and tests the basename
  (`config.go:792`, `engine.go:156`). Its anchoring is wrong for prose: text
  rules are about a pattern appearing (or not) *somewhere* in a span, not the
  span matching exactly. The text family reuses the `pattern` config key and the
  `AnchoredPattern` precedent but compiles **unanchored**.
- **Four-touchpoint wiring, parity-enforced.** A new check type is a constant +
  `normalizeCheck` case in `internal/config/config.go`, a `Run` in
  `internal/checks/`, a dispatch case in `cmd/engine.go` (`checksFor`), and a
  `Descriptor` in `registry.go`; `registry_test.go` fails the build if a
  dispatched type has no Descriptor, and `cmd/gendocs` renders
  `docs/content/reference/check-types/` from the Descriptors. The
  `add-katalyst-check-type` skill documents the sequence.
- **Checks are per-item and check-only.** Text rules need no sibling context, so
  they implement the existing per-item `Check.Run(ctx) []Violation` contract;
  they emit `Violation`s and never auto-fix (consistent with `fix` refusing
  semantic edits, `fix.go`).

## Design

### A fourth family: `text`

Add a `text` family alongside `objects`/`markdown`/`filesystem`. The split from
`markdown` is deliberate and load-bearing for the docs taxonomy: markdown checks
are markdown-*aware* (they parse headings and fences and only make sense on
markdown), while text checks are structure-agnostic — they operate on raw lines
and the whole body string and apply equally to a `.txt` file. Folding them into
`markdown` would force the markdown family intro ("relationships between
frontmatter metadata and markdown body content") to also mean "arbitrary regex
over raw text," muddying both the intro and the reference grouping. New family
intro: *"Text check types validate raw body content with regex and string-policy
rules, independent of markdown structure."*

### Two axes: target × rule

Like the filesystem family's `target × rule` model, every text check answers
*which span of text* and *what must hold of it*.

**Target** — selected by a `target` key, default `body`:

| `target` | Value(s) tested |
|---|---|
| `body` | the entire body as one multiline string (newlines preserved) |
| `line` | each body line, evaluated independently |
| `first-line` | the first non-empty body line only |
| `matched-lines` | each body line matching the `select:` regex, evaluated independently |

`body` is the document-level span (a multiline regex can span lines); the other
three are line-level and yield one diagnostic per offending line. `first-line`
skips leading blank lines so a rule about "the opening line" is not defeated by a
blank line after the frontmatter fence. `matched-lines` is the escape hatch for
"heading lines" or "matched regions" from the issue — e.g. `select: '^#'` to
target ATX headings without the text family having to know what a heading is.

### Two rules

| Kind | Fields | Meaning |
|---|---|---|
| `text_requires` | `pattern` (req), `target`, `select`, `literal` | The pattern must be **present** in the target. |
| `text_forbids` | `pattern` (req), `target`, `select`, `literal` | The pattern must be **absent** from the target. |

`pattern` is a Go regular expression, compiled **unanchored** (a deliberate
divergence from `filesystem_name_regex`, which anchors — see Current State).
`literal: true` escapes `pattern` with `regexp.QuoteMeta` so it matches as a
plain substring, covering the issue's "string-policy" half without a separate
kind. `select` is only valid with `target: matched-lines`.

Presence/absence semantics per target:

- **`target: body`** — `text_requires` passes if the pattern matches anywhere in
  the body; `text_forbids` fails on the first match.
- **`target: first-line`** — the rule is evaluated against that one line.
- **`target: line` / `matched-lines`** (multi-value) — the rule is evaluated
  **per selected line**: `text_forbids` emits one violation for *each* line that
  matches (e.g. every line ending in a period); `text_requires` emits one
  violation for *each* selected line that does **not** match. So "every heading
  line must start with a capital letter" is `text_requires`, `target:
  matched-lines`, `select: '^# '`, `pattern: '^# [A-Z]'`. (Whether
  `text_requires` over a multi-line target should mean "every line" or "at least
  one line" is **Open Question 1**.)

### Diagnostics

Every violation carries the file and a 1-based line number.

- Line-level targets report the line directly (`BodyLine + index`, exactly as
  `markdownLines` already computes).
- For `target: body`, the match can start anywhere. Compute the violation line
  by counting newlines in `Body` up to the match's start offset and adding
  `BodyLine`. `text_forbids` reports the first match; `text_requires` reports
  line `BodyLine` (start of body) since "missing" has no offset. The `Path`
  field stays `/` (body-scoped, no frontmatter pointer), matching how the
  markdown family reports body violations (`markdown.go:36`).

Messages follow the check diagnostic grammar (`path:line: message`): e.g.
`forbidden text /\bTODO\b/ found` and `required text /Generated by/ not found`,
quoting the pattern with the regex delimiters so the source of the failure is
legible.

### Config surface

`internal/config/config.go`:

- Add `CheckTextRequires` and `CheckTextForbids` constants.
- Extend `rawCheck` and `CheckInstance` with a `Select string` and `Literal
  bool` (reuse the existing `Pattern` and `Target` fields).
- `normalizeCheck` cases: require `pattern`; compile it unanchored to reject
  invalid regexes at load time (as `filesystem_name_regex` does eagerly,
  `config.go:796`); validate `target` against `{body, line, first-line,
  matched-lines}`; require `select` **iff** `target: matched-lines` and reject it
  otherwise; default `target` to `body`.

`internal/checks/text.go` (new): `TextRequires` / `TextForbids` structs with a
shared `selectSpans(ctx, target, select) []span` helper (each span carries text +
line) and a compiled `*regexp.Regexp`. `cmd/engine.go` `checksFor` dispatches the
two new types, compiling `pattern` via a new unanchored helper beside
`AnchoredPattern`.

### `fix` interaction

Text checks are **check-only**. Even a mechanical fix (strip a trailing period,
delete a forbidden word) edits authored prose, which `fix` refuses on the same
grounds it won't invent frontmatter values. Violations are reported with the
offending line so the author edits deliberately.

## Open Questions

1. **`text_requires` quantifier over multi-line targets.** With `target: line`
   or `matched-lines`, does "requires" mean *every* selected line must match
   (current Design) or *at least one* line in the document matches? "Every" makes
   `text_requires`/`text_forbids` clean duals (forbids = no line matches, requires
   = no line fails); "at least one" serves "the document must contain a line like
   X." Recommendation: ship "every" for line targets and express "at least one"
   as `text_requires` over `target: body` with a multiline pattern, so each kind
   has one meaning. Needs confirmation before finalizing.

2. **`literal` flag vs a separate substring kind.** Fold string-policy into the
   regex kinds via `literal: true` (current Design), or add explicit
   `text_excludes_substring` / `text_includes_substring` kinds taking a list of
   plain strings? The flag keeps the family at two kinds; a list-valued substring
   kind reads better for "ban this set of words" but duplicates the rule axis.
   Recommendation: ship the `literal` flag; revisit a list form if authors ask.

3. **Scope of `matched-lines`/`select` in v1.** Shipping `select` adds a second
   user-facing regex per check and a validation branch. Defer it (ship only
   `body`/`line`/`first-line`) or include it? The issue explicitly lists "heading
   lines, matched regions," which only `matched-lines` serves without the family
   learning markdown structure. Recommendation: include it; it is the feature's
   stated reason for existing.

## Test checklist (what the pending tests assert)

- [ ] `text_forbids target: body` flags a pattern present in the body; passes
      when absent
- [ ] `text_requires target: body` passes when present; flags when absent
- [ ] `target: first-line` evaluates the first non-empty line and skips leading
      blanks
- [ ] `target: line` `text_forbids` emits one violation per offending line, each
      with the correct line number
- [ ] `target: matched-lines` with `select` evaluates only matching lines
- [ ] `select` is rejected unless `target: matched-lines`; `target:
      matched-lines` requires `select`
- [ ] `literal: true` matches the pattern as a plain substring (regex metachars
      inert)
- [ ] patterns are unanchored (a mid-string match counts)
- [ ] an invalid `pattern` is a load-time error
- [ ] `target: body` violations report the line of the match (newline count +
      `BodyLine`)
- [ ] a frontmatter-less file is linted (Body = whole file, lines from 1)
- [ ] every new kind has a Descriptor (registry parity stays green)
- [ ] `make docs-gen` produces the `text` family reference cleanly
