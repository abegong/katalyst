# Text checks: regex and string policy over body content

> **Status: planning.** Adds text rules that run regex/string-policy checks
> directly against an item's body (and against plain-text files with no
> frontmatter), with line-level diagnostics, optional auto-fixes, and inline
> examples. Resolves issue #49. **Open structural question:** whether these ship
> as a new fourth `text` family or are arranged some other way ŌĆö parked for
> discussion (Open Question 1); the rest of the design is independent of it.

## Overview

Katalyst checks today reach frontmatter (the `object` family), markdown
structure (the `markdown` family), and filename/path conventions (the
`filesystem` family). Nothing lints the **body prose** as raw text: "no trailing
period on the title line," "must contain the words *Generated by*," "never write
*TODO* in a published note." These are presence/absence rules over text, and the
only check that touches the body ŌĆö the markdown family ŌĆö reasons about headings
and fences, not arbitrary line shapes. This spec adds two rules, `text_requires`
and `text_forbids`, that target the body directly: a span selector (`body`,
lines, paragraphs, lines relative to a match), a regex/substring matcher, an
optional declared **fix**, and inline **examples** that double as documentation
and self-tests.

## Value

- **Prose constraints become first-class.** Style rules ("no trailing
  whitespace," "title line has no period," "forbidden phrasing") are the rules
  authors hit first in a writing vault, and today they need external linters or
  brittle wrappers. One documented surface (`pattern`, `target`) covers them.
- **Plain-text files are in scope, not just frontmatter-bearing markdown.** The
  parser already returns the entire file as `Body` when there is no frontmatter
  (`frontmatter.go:138`), so a collection of `.txt` or frontmatter-less `.md`
  files can be linted the same way ŌĆö the rule just needs to read `Body`.
- **Fixable where it's safe.** Many prose violations (a stray trailing period, a
  forbidden synonym) are a mechanical search-and-replace. Letting a rule declare
  its own fix makes `katalyst fix` repair them, extending `fix` from a
  frontmatter formatter to a body repairer **only where the author opts in**.
- **Examples make rules legible and self-testing.** A rule that ships positive
  and negative examples documents itself and can be verified ŌĆö including that its
  fix turns each negative example into the stated result.

## Current State

- **Three families, none reads raw body text.** `internal/checks/registry.go`
  defines `objects`, `markdown`, and `filesystem` families (`Families()`).
  `object`/`filesystem` never look at the body; the `markdown` family does, but
  only through structure-aware helpers (`firstH1`, `heading`, fence scanning in
  `internal/checks/markdown.go`).
- **The body is already available, line-numbered.** Every check receives
  `Context{Doc *frontmatter.Document, ŌĆ”}` (`internal/checks/checks.go:6`).
  `Doc.Body []byte` holds the body and `Doc.BodyLine` is the 1-based line where
  the body starts (`frontmatter.go:100`). The markdown family already splits the
  body into `(line, text)` pairs via `markdownLines(body, bodyLine)`
  (`markdown.go:171`); text rules reuse that exact helper for per-line
  diagnostics. For a file with no frontmatter, `Body` is the whole file and
  `BodyLine` is 1 (`frontmatter.go:138`), so line numbers stay correct.
- **A regex check type already exists, but anchored and over the filename.**
  `filesystem_name_regex` compiles `^(?:pattern)$` and tests the basename
  (`config.go:792`, `engine.go:156`). Its anchoring is wrong for prose: text
  rules are about a pattern appearing (or not) *somewhere* in a span, not the
  span matching exactly. Text rules reuse the `pattern` key and the
  `AnchoredPattern` precedent but compile **unanchored**.
- **`fix` is a frontmatter formatter that refuses to invent.** `cmd/fix.go`
  canonicalizes frontmatter (sorted keys, block style, one trailing newline) and
  **preserves the body verbatim**; it "never invents semantic values." A
  rule-declared body fix is a deliberate, scoped extension of this contract ŌĆö see
  Design.
- **Four-touchpoint wiring, parity-enforced.** A new check type is a constant +
  `normalizeCheck` case in `internal/config/config.go`, a `Run` in
  `internal/checks/`, a dispatch case in `cmd/engine.go` (`checksFor`), and a
  `Descriptor` in `registry.go`; `registry_test.go` fails the build if a
  dispatched type has no Descriptor, and `cmd/gendocs` renders
  `docs/content/reference/check-types/` from the Descriptors.
- **Checks are per-item.** Text rules need no sibling context, so they implement
  the existing per-item `Check.Run(ctx) []Violation` contract.

## Design

### Family placement (parked)

Where these rules live in the docs taxonomy ŌĆö a new fourth `text` family, or
some other arrangement ŌĆö is **Open Question 1** and is being discussed
separately. The rest of this Design (span selector, the two rules, fix,
examples) holds regardless of that answer; the family decision changes only the
`Family` value in the registry and the reference subdirectory the pages render
into. The spec provisionally writes them as a `text` family so the examples read
concretely.

### The span selector: `target`

Every text rule is evaluated against a set of **spans** ŌĆö slices of body text ŌĆö
chosen by `target` (default `body`). A rule with one span (`body`, `first-line`)
is a single assertion; a rule whose target yields many spans (`line`,
`paragraph`, `matched-lines`) is evaluated per span.

| `target` | Spans produced |
|---|---|
| `body` | one span: the entire body as a multiline string |
| `line` | one span per body line |
| `first-line` / `last-line` | one span: the first/last non-empty body line |
| `line-number` (+ `index: N`) | one span: the Nth body line (1-based; negative counts from the end) |
| `paragraph` | one span per blank-line-delimited block (multiline) |
| `matched-lines` (+ `select: <regex>`) | one span per body line matching `select` |

Two modifiers compose with the line targets:

- **`index: N`** picks the Nth span for an indexed target ŌĆö e.g.
  `target: paragraph, index: 1` is the first paragraph; `target: line-number,
  index: -1` is the last line.
- **`offset: N`** combines with `matched-lines` to select the line *N* lines
  after (negative = before) each matched line instead of the matched line
  itself. "The line after every heading" is `target: matched-lines, select: '^#
  ', offset: 1`. This is the "nth line before/after a matched line" capability:
  it lets a rule about a *related* line (the line under a heading, the line
  before a fence) be written without the family learning markdown structure.

`first-line` skips leading blank lines so an "opening line" rule is not defeated
by a blank line after the frontmatter fence. `select` and `offset` are valid only
with `matched-lines`; `index` only with `line-number`/`paragraph`. (How much of
this selector ships in v1 is **Open Question 2**.)

### The two rules: `text_requires` and `text_forbids`

Two kinds carry the polarity in their name. Both match `pattern` (a Go regex,
compiled **unanchored** ŌĆö the deliberate divergence from `filesystem_name_regex`,
whose `^ŌĆ”$` anchoring is wrong for prose) against each span the `target` selects.
`literal: true` wraps `pattern` in `regexp.QuoteMeta` so it matches as a plain
substring, covering the issue's "string-policy" half (**Open Question 3**).

- **`text_forbids`** ŌĆö the pattern must appear in **no** span. "No line contains
  TODO" is `kind: text_forbids, target: line, pattern: '\bTODO\b'`. It needs no
  further knob: "forbid X" only ever sensibly means "X is in no span."
- **`text_requires`** ŌĆö the pattern must appear, with a `match` knob (default
  `any`) that only does work for multi-span targets:

  | `match` | Passes when |
  |---|---|
  | `any` (default) | at least one selected span matches ŌĆö "the body mentions *Sources*" |
  | `all` | every selected span matches ŌĆö "every heading line is Title Case" |

  For single-span targets (`body`, `first-line`) `any` and `all` coincide, so the
  knob matters only with `line`/`paragraph`/`matched-lines`. `match: any | all`
  is spelled as two self-describing values rather than a boolean so neither the
  field nor a magic default needs decoding.

Diagnostics name the failing span(s): `text_forbids` reports each matching span;
`text_requires match: all` reports each non-matching span; `text_requires match:
any` reports one body-level violation when nothing matched.

### Optional fix

Only `text_forbids` may declare a `fix` ŌĆö a replacement template applied to the
matched (forbidden) text with `regexp` capture-group syntax (`$1`, `${name}`):

```yaml
- kind: text_forbids
  target: first-line
  pattern: '\.(\s*)$'   # a period (+ trailing space) ending the first line
  fix: '$1'             # drop the period, keep any trailing whitespace
```

The asymmetry is principled. `text_forbids` always **locates** the offending
text ŌĆö that is exactly what its pattern matched ŌĆö so a fix has a concrete span to
rewrite. `text_requires` has nothing to locate: a missing required phrase has no
position, and inventing one is the value-invention `fix` already refuses
(`cmd/fix.go`). So a `fix` on `text_requires` is a config error. Further:

- **A fix replaces only the matched substring** within the span, never the whole
  span or file, so unrelated text is untouched. The replacement is purely
  mechanical (regex substitution), like the rest of `fix`.
- **`fix` is opt-in.** `text_forbids` rules without `fix` stay report-only,
  exactly as today; the `fix` command's body-preservation default is unchanged
  for them.

The engine applies the replacement generically from the config (pattern +
replacement + span), so no per-check-type fix code is needed. Whether to re-run
the rule after applying (catching a bad template) is **Open Question 4**.

### Examples

A rule may carry `examples` ŌĆö positive and negative samples that document it and
are machine-verified:

```yaml
- kind: text_forbids
  target: first-line
  pattern: '\.(\s*)$'
  fix: '$1'
  examples:
    pass:
      - "My Note Title"
    fail:
      - text: "My Note Title."
        fixed: "My Note Title"      # required iff the rule declares a fix
```

Verification asserts: every `pass` example satisfies the rule; every `fail`
example violates it; and when the rule declares a `fix`, applying it to each
`fail.text` yields `fail.fixed` (which then passes). This makes examples
executable documentation ŌĆö the same samples render into the generated reference
page. The mechanism is generic over check types, not text-specific. How far to
take it in v1 ŌĆö required vs optional, where it is verified, and whether to defer
the whole feature ŌĆö is **Open Question 5**.

### Diagnostics

Every violation carries the file and a 1-based line number.

- Line/paragraph spans report their own start line (`BodyLine + index`, as
  `markdownLines` already computes).
- For `target: body`, compute the line by counting newlines in `Body` up to the
  match's start offset and adding `BodyLine`. The `Path` field stays `/`
  (body-scoped), matching how the markdown family reports body violations
  (`markdown.go:36`).

Messages follow the `path:line: message` grammar: `forbidden text /\bTODO\b/
found`, `required text /Sources/ not found`, quoting the pattern with regex
delimiters.

### Config surface

`internal/config/config.go`:

- Add `CheckTextRequires` and `CheckTextForbids` constants.
- Extend `rawCheck`/`CheckInstance` with `Match string` (requires only),
  `Select string`, `Index *int`, `Offset *int`, `Literal bool`, `Fix string`,
  and an `Examples` block (`pass []string`, `fail []{text, fixed}`). `Pattern`
  and `Target` are reused.
- `normalizeCheck`: require `pattern`; compile unanchored to reject bad regexes
  at load time (as `filesystem_name_regex` does eagerly, `config.go:796`);
  validate `target`; validate `match Ōłł {any, all}` and reject it on
  `text_forbids`; require `select` iff `matched-lines` and reject
  `offset`/`select`/`index` outside their targets; reject `fix` on
  `text_requires`; require `fail.fixed` iff `fix` is set.

`internal/checks/text.go` (new): `TextRequires`/`TextForbids` structs sharing a
`spans(ctx) []span` helper (each span carries text + start line) and a compiled
`*regexp.Regexp`. `cmd/engine.go` dispatches them; `cmd/fix.go` gains a body pass
that applies declared replacements. `registry.go` gets the Descriptors and, per
Open Question 1, possibly a fourth `Family`.

## Open Questions

### 1. Family placement (structural; parked for discussion)

**Context.** Check types are grouped into *families* ŌĆö `objects`, `markdown`,
`filesystem` ŌĆö defined by `Families()` in `internal/checks/registry.go`. A
family fixes the docs subdirectory (`reference/check-types/<family>/`), the intro
copy, and how `katalyst check-types list` groups output. Text rules need a home.
The `markdown` family already reads the body, but only through markdown-*aware*
helpers (headings, fences); text rules are structure-agnostic and also apply to
non-markdown `.txt` files.

**Choices & tradeoffs.**

| Option | Buys | Costs |
|---|---|---|
| New `text` family | A truthful intro ("raw body content, independent of markdown structure"); honest that rules apply to `.txt` | A fourth family is more surface for users to learn and docs to maintain |
| Fold into `markdown` | No new family; all body checks in one place | Forces the markdown intro to also mean "arbitrary regex over raw text," blurring markdown-aware vs structure-agnostic, and misreads for `.txt` |
| Rename to a broader family (`content`) spanning markdown + text | One body-content family | Renames existing markdown pages/kinds ŌĆö real churn for a taxonomy bet |

**Recommendation.** New `text` family: the structure-agnostic vs markdown-aware
distinction is real and keeps each family intro true. Your call.

### 2. How much of the `target` selector ships in v1

**Context.** The Design's selector offers `body`, `line`,
`first-line`/`last-line`, `line-number` (+`index`), `paragraph`, and
`matched-lines` (+`select`, +`offset`). Each target/modifier is config keys,
validation, and tests. Issue #49 names only full-document text, specific lines
(first line, heading lines), and matched regions; the nth-line, paragraph, and
relative-line targets came from review feedback.

**Choices & tradeoffs.**

| Option | Buys | Costs |
|---|---|---|
| Full set now | nth-line, paragraph, and relative-to-match all available in v1; matches the feedback | Most config surface, validation, and tests up front; `paragraph`/`offset` may sit unused initially |
| Core set (`body`, `line`, `first-line`, `matched-lines`+`select`); defer `line-number`, `last-line`, `paragraph`, `offset` | Satisfies #49's stated asks with the least surface | A later additive key-add when authors reach for the rest (no model change, low risk) |

**Recommendation.** Lean **full set**, since you specifically asked for nth-line,
relative-line, and paragraph targets ŌĆö they share one `spans()` helper, so the
marginal cost is per-target parsing/tests, not new architecture. If you'd rather
keep v1 lean, the core set is a clean cut and the rest is purely additive later.
Your call.

### 3. Literal/substring matching: flag vs dedicated list kind

**Context.** Beyond regex, #49 asks for "string-policy" checks ŌĆö matching plain
literal strings, where regex metacharacters should be inert. Two ways to spell
it. A `literal: true` boolean treats `pattern` as a literal substring (via
`regexp.QuoteMeta`). A dedicated list-valued kind bans a *set* of literals in one
check, e.g. `kind: text_forbids_substrings, substrings: [TODO, FIXME, XXX]`.

**Choices & tradeoffs.**

| Option | Buys | Costs |
|---|---|---|
| `literal: true` flag on the two rules | One mechanism, one code path; a word-set is a regex alternation `TODO\|FIXME\|XXX` | Banning many literals means authoring/escaping an alternation; metachars in a literal require remembering the flag |
| Separate list-valued substring kind(s) | Reads cleanly for word-bans; no regex knowledge needed | Duplicates the rule axis ŌĆö its own `target`/`fix`/`examples` handling and Descriptors; two ways to do one thing |
| Both | Convenience and power | Most surface; overlapping kinds to document and keep consistent |

**Recommendation.** Ship the `literal: true` flag and lean on alternation for
sets; add a list kind only if word-ban configs prove common in practice. Your
call.

### 4. Should `fix` re-check its own work?

**Context.** A `text_forbids` rule can declare a `fix` (a regex replacement). A
badly written template could rewrite the match yet leave the document still
violating ŌĆö e.g. the replacement re-introduces the forbidden pattern, or matches
in a way the author didn't foresee. The question is whether `katalyst fix`
verifies the result.

**Choices & tradeoffs.**

| Option | Buys | Costs |
|---|---|---|
| Re-run the rule after applying; error if still violating | Catches bad templates at fix time; `fix` never silently no-ops | One extra check pass over fixed items |
| Trust the rule's `fail.fixed` examples (OQ5) to have caught it offline | No extra pass | A template that is correct on its examples but wrong on real text fails silently |

**Recommendation.** Re-check after fixing and surface "fix did not resolve the
violation." A silent no-op fix is worse than today's frontmatter-only `fix`, and
the extra pass is cheap (the item is already in memory). Your call.

### 5. Examples: scope in v1

**Context.** A rule may carry `examples` (pass/fail samples, with `fixed` results
when a fix is declared) that both document the rule and can be machine-verified.
This is a new, cross-cutting mechanism ŌĆö it could apply to every family, not just
text ŌĆö and you flagged it as something to *explore*, so its scope is genuinely
open. Three independent axes:

| Axis | Options |
|---|---|
| Required? | optional per rule **vs** every text rule must ship examples |
| Verified where? | a `katalyst check-types verify` command + CI **vs** also enforced at config **load** time (examples must hold before the rule runs) **vs** docs-only (not machine-verified) |
| Scope | wired generically for all families now **vs** text-only in v1 **vs** defer the whole examples feature to a follow-up spec |

**Recommendation.** Optional; verified by an explicit `check-types verify`
command + CI (not load time ŌĆö that would slow every `check`); built generic but
documented for text rules only in v1. If examples feel like scope creep for this
spec, deferring them to a follow-up is clean ŌĆö they don't change the rule model.
Your call.

## Test checklist (what the pending tests assert)

Span selector:
- [ ] `body` matches across newlines (multiline pattern)
- [ ] `line` evaluates each line; `first-line`/`last-line` skip blanks
- [ ] `line-number` with `index: N` (incl. negative) selects the right line
- [ ] `paragraph` splits on blank lines; `index` picks the Nth
- [ ] `matched-lines` + `select` selects matching lines; `offset` shifts to the
      line N before/after each match
- [ ] modifier misuse is a load error (`select`/`offset` off `matched-lines`,
      `index` off an indexed target)

Rules:
- [ ] `text_forbids` fails when the pattern is in any span; passes when absent
- [ ] `text_requires match: any` passes on one match; `match: all` requires every
      span; `any`==`all` for `body`/`first-line`
- [ ] `match` is rejected on `text_forbids`
- [ ] `literal: true` matches metachars inertly; patterns are unanchored
- [ ] an invalid `pattern` is a load-time error
- [ ] `body` violations report the match's line; line/paragraph spans report
      their own line; a frontmatter-less file lints with lines from 1

Fix:
- [ ] a declared `fix` rewrites only the matched substring; document re-checks
      clean (or errors per OQ4)
- [ ] `fix` on `text_requires` is a load error
- [ ] `text_forbids` without `fix` stays report-only; `fix` still preserves body

Examples:
- [ ] each `pass` example satisfies the rule; each `fail` violates it
- [ ] when `fix` is set, applying it to `fail.text` yields `fail.fixed`; `fixed`
      is required iff `fix` is set

Registry / docs:
- [ ] every new kind has a Descriptor (parity stays green)
- [ ] `make docs-gen` renders the reference cleanly
