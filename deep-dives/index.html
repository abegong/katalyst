    Deep dives#

    Understanding-oriented discussion of the why behind Katalyst, the vision and scope, the core concepts the tool is built on, the domain model that instantiates them in katalyst, and the deeper design discussions that no single page or package owns: how the core commands are organized, how the storage layer maps stores onto the model, and how operations grow richer as a backend’s capabilities increase. For the short version, start with Why Katalyst.

    Rationale that is tied to a specific subsystem lives next to the code, in the per-package README.md files under internal/ (for example internal/config, internal/frontmatter, and internal/checks). The pages here cover the cross-cutting why that no single package owns.
